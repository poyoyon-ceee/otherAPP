1. そもそもGitって？なぜ使うの？

Gitは、ファイルの変更履歴を記録・管理するツールです。一言で言えば、コードの「タイムマシン」のようなもの。これがあれば、いつでも過去の状態に戻したり、複数人で安全に共同作業を進めたりできます。

Gitを理解する2つのカギ

用語 説明 イメージ
ローカルリポジトリ あなたのPC内にある、Gitが管理するプロジェクトフォルダ。全ての変更履歴がここに保存され、オフラインでも作業できます。 自分の机
リモートリポジトリ GitHubなどのオンライン上にある、コードを共有するための「中央保管庫」。ここに変更をアップロード（プッシュ）したり、ダウンロード（プル）したりします。 共有の本棚

2. 最初の準備：初期設定（初回のみ）

GitとGitHubが正しく連携できるように、PCごとに一度だけ以下の設定を行います。新しいPCで開発を始めるときの必須作業です。

ステップ 2.1: Gitに自分の名前を教える

目的: あなたが行ったすべての変更（コミット）に、「誰が変更したか」というサインを残すためです。これにより、共同作業時に誰がどのコードを書いたかが一目で分かります。

よくあるエラー: Git の "user.name" と "user.email" を構成していることを確認してください。

解決策: Cursorのターミナルで、以下の2つのコマンドを実行してください。一度設定すれば、PCに保存されます。

git config --global user.name "あなたのGitHubユーザー名"
git config --global user.email "あなたのGitHub登録メールアドレス"
💡ヒント: --global は、このPCで行うすべてのGit作業にこの設定を適用するという意味です。

ステップ 2.2: PCとGitHubを紐づける

目的: あなたのPC（ローカルリポジトリ）が、どのGitHubリポジトリ（リモートリポジトリ）とデータをやり取りするのかをGitに教えるためです。この紐付けがないと、変更のアップロードやダウンロードができません。

よくあるエラー: fatal: 'origin' does not appear to be a git repository

解決策:

  1. GitHubで、対象のリポジトリページを開きます。
  2. 緑色の Code ボタンをクリックし、HTTPSタブに表示されるURLをコピーします。
  3. Cursorのターミナルで以下のコマンドを実行します。
git remote add origin <GitHubでコピーしたURL>
💡ヒント: origin は、紐付けたリモートリポジトリを指すためのニックネームのようなものです。一般的にこの名前が使われます。

3. 日常の作業サイクル：変更を記録し、共有する

日々のコーディング作業で、Gitをどのように活用するかをCursorの便利なGUI（グラフィカル・ユーザー・インターフェース）操作で見ていきましょう。

ステップ 3.1: 変更を記録する（ステージング → コミット）

1. ステージング：記録する変更を選ぶ
「ステージング」とは、変更したファイルの中から、一つのまとまりとして記録（コミット）したいものだけを選ぶ準備作業です。

  • Cursorの左サイドバーにあるソース管理アイコン（分岐マーク）をクリックします。
  • 変更したファイルの一覧が表示されます。記録したいファイルの横にある + ボタンを押して、「ステージされている変更」エリアに移動させます。

2. コミット：変更にメッセージを付けて記録する
「コミット」は、ステージングした変更をローカルリポジトリの歴史に刻む作業です。

  • 上部のメッセージボックスに、「何のために、何をどう変更したか」が簡潔にわかるメッセージを入力します。（例：「ログイン機能のバグを修正」）
  • コミット ボタンをクリックします。これで、あなたのPCに変更履歴の「セーブポイント」が作成されました。

ステップ 3.2: 変更を共有する（プッシュ ＆ プル）

ローカルリポジトリ（あなたのPC）とリモートリポジトリ（GitHub）の状態を常に最新に保つことが、スムーズな開発の秘訣です。

操作 説明
プッシュ (Push) ローカルでコミットした変更を、GitHubにアップロードします。
プル (Pull) 他の人がGitHubにアップロードした最新の変更を、自分のPCにダウンロードします。

一番簡単な方法：同期ボタンを使おう！
Cursorの画面下にあるステータスバーの雲のアイコンか、ソース管理パネル上部にある更新アイコンをクリックします。このボタン一つで、プルとプッシュを賢く同時に実行してくれます。

CursorでのGit操作画面の実際の使い方

📁 ソース管理パネルの構成要素
  • 変更されたファイル: 編集したファイルが一覧表示される
  • ステージされた変更: コミット準備ができたファイル
  • メッセージ入力欄: コミットメッセージを記入
  • コミットボタン: 変更をローカルリポジトリに記録
  • 同期ボタン: GitHubとの同期（プッシュ・プル）
🔄 実際の操作手順
  1. ファイルを編集: コードを変更すると、ファイル名が「変更」セクションに表示される
  2. ステージング: ファイル名の横の「+」ボタンをクリックして「ステージされた変更」に移動
  3. コミットメッセージ入力: 「何を変更したか」を簡潔に記述
  4. コミット実行: 「コミット」ボタンでローカルに変更を記録
  5. GitHubに同期: 「同期」ボタンでリモートリポジトリにアップロード
💡 実践的なTips
  • コミットメッセージは「動詞 + 目的語」で書く（例：「ログイン機能を修正」）
  • 関連する変更はまとめて1つのコミットにする
  • 作業前に必ず「同期」して最新状態を確認
  • エラーが出た場合は、ターミナルで詳細を確認

GitとGitHubの関係性

Gitは、あなたのPC上で動作するバージョン管理ツールです。一方、GitHubは、インターネット上のリポジトリホスティングサービスです。この2つは役割が異なります。

なぜGitHubに先にリポジトリを作成するのか？

ローカルでgit initを実行すると、あなたのPC内に隠しフォルダ（.git）が作成され、ファイルの変更履歴が記録され始めます。しかし、この時点ではまだGitHubとは一切関係がありません。

git remote add originコマンドは、「ローカルの.gitフォルダを、GitHubのこのURLと紐づけますよ」と設定するために使います。したがって、紐づけ先のGitHubリポジトリが事前に存在している必要があります。

たとえるなら、GitHubに「ファイルの保管場所」を事前に用意しておき、ローカルからその場所にファイルを送り届ける、というイメージです。

4. 困ったときは：よくあるトラブル解決集

Git初心者が遭遇しやすいエラーとその解決策をまとめました。焦らず、原因と対処法を確認しましょう。

トラブル 4.1: プッシュが拒否される！

エラーメッセージ:
! [rejected], non-fast-forward
hint: Updates were rejected because the remote contains work that you do not have locally.
  • 原因: あなたが知らないうちに、他の誰かがGitHubに変更をプッシュしました。あなたのPCの状態が古くなっているため、Gitが「他人の変更を上書きしてしまう危険がある」と判断し、プッシュを止めてくれています。
  • 対処法:
    1. まずGitHubの最新の変更をプル（ダウンロード）します。
    2. その後、改めてプッシュします。
git pull origin main
git push origin main

トラブル 4.2: プルしようとすると「無関係な歴史」と言われる！

エラーメッセージ: fatal: refusing to merge unrelated histories
  • 原因: ローカルとリモートのリポジトリが、それぞれ別々に作られてしまい、共通の歴史（コミット）を持っていない状態です。
  • 対処法: 以下のコマンドで、2つの異なる歴史を強制的に関連付けて統合します。
git pull origin main --allow-unrelated-histories
⚠️注意: このコマンドは強力ですが、意図しないファイルのマージが発生する可能性もあります。通常はプロジェクトの最初に一度だけ使います。

トラブル 4.3: ファイルが「コンフリクト（競合）」した！

エラーメッセージ: CONFLICT (add/add), Automatic merge failed; fix conflicts and then commit the result.
  • 原因: あなたと他の人が、同じファイルの同じ部分を同時に変更してしまいました。Gitがどちらの変更を優先すべきか自動で判断できない状態です。
  • 対処法: 手動で正しいコードを選び、Gitに教えてあげる必要があります。
    1. コンフリクトが起きているファイルをCursorで開きます。<<<<<<<, =======, >>>>>>> という特殊なマーカーで競合箇所が示されています。
    2. Cursorのエディタ上に表示される Accept Current Change（自分の変更を採用）や Accept Incoming Change（相手の変更を採用）などのボタンを使い、残したいコードを選択します。もしくは、マーカーごと手動でコードを編集して、最終的に正しい形にします。
    3. ファイルを保存した後、以下のコマンドで解決したことをGitに伝えます。
# 1. 解決したファイルをステージング
git add .
# 2. 解決したことをコミット
git commit -m "マージコンフリクトを解決"
# 3. GitHubにプッシュ
git push origin main
🚫最終手段: どうしても解決できない場合、git merge --abortでマージを中止し、git push -f origin mainで強制的に自分の変更を上書きできます。しかし、他人の変更を消してしまう危険な操作なので、実行する前に必ずチームに相談してください。

🚨 トラブル 4.4: 特殊な操作：コミット履歴をすべて消去する（危険）

⚠️ 重要警告: この操作はすべてのコミット履歴を完全に消去します。実行前に必ずバックアップを作成し、チームメンバーと相談してください。

目的: プロジェクトの過去のコミット履歴をすべて消し去り、現在の状態だけを「最初のコミット」として残す特殊なコマンドです。主に以下のような場合に使用されます：

  • 大きなファイルや機密情報が履歴に含まれてしまった場合
  • プロジェクトを完全にリセットしたい場合
  • 履歴をクリーンにしたい場合
git checkout --orphan tmp
git commit -m "Initial Commit"
git checkout -B main
git branch -d tmp
🔍 コマンド解説
  • git checkout --orphan tmp: 親を持たない、完全に独立したブランチを作成
  • git commit -m "Initial Commit": 最初のコミットを作成
  • git checkout -B main: 古いmainブランチを強制的に上書き
  • git branch -d tmp: 作業用ブランチを削除
🚨 実行前のチェックリスト
  • ✅ 現在のコードが正常に動作することを確認
  • ✅ 重要なファイルがすべて含まれていることを確認
  • ✅ チームメンバー全員に事前通知
  • ✅ リモートリポジトリのバックアップを作成
  • ✅ 実行後にgit push -f origin mainで強制プッシュが必要
